# Anatomia — コード解析 × 動作構造 × 決定的キャッシュ

> 作業名 **Anatomia** (仮称)。コードを「機構」へ解剖し、仕様と実行時挙動に結びつけ、
> 決定的キャッシュの上で可視化する開発者ツール。正式名・略称は `project-codes` で別途確定する。
>
> 本書は方針確定時点の設計 (実装は未着手)。形態 (Ergo plugin / 独立サービス / lib) と
> 重心 (AI 文脈供給 / 人間可視化) は §13 に未決定として残す。

---

## 1. 目的とスコープ

ゲームのコードベースを、次の 2 用途に同時に使える知識構造へ変換する。

- **(a) AI への文脈供給**: 高精度・キャッシュ安定な「機構単位の文脈束」を決定的に組んで渡す。
- **(b) 人間の把握**: 動作構造 (メインループ / システム実行順) と複雑度を可視化する。

本質は **静的構造 (何の機構が存在するか) と 動的実行 (今フレーム何が火を噴くか) を、
同一のアンカー ID で縫合すること**。これにより「仕様 ↔ コード ↔ 実行」の三層が一枚になる。

**主対象**: C++/C# のゲーム基盤 (Pictor/Ergo 系・Unity 系)。
**副対象**: 肥大化した非ゲーム (Memoria 等) にも汎用オントロジーで適用可。

非対象: プロンプト内容のチューニング、モデル選定、本ツール外のビルド/CI 統合。

---

## 2. 設計思想 (3 原則)

1. **キャッシュをデータ構造にする。** 後付けの索引でなく、コンテンツアドレッシング (Merkle) で
   無効化を構造から *無料で・正確に* 得る。ビルドキャッシュ (Bazel/Nix) や git と同じ原理。
2. **静的と動的を同一アンカー ID で縫う。** 「存在する機構」と「実行された機構」が同じ ID で繋がる。
3. **結合は類似度でなく適合・証拠で。** embedding は補助信号に留め、契約 (conformance) と
   明示参照を一級市民にする。

> 重要な一致: 本ツールのキャッシュ無効化 (AST ノード変更 → 祖先を無効化) は、
> LLM プロンプトキャッシュの無効化 (トークン変更 → 後続を無効化) と **同じ法則**。
> 前方一致が「部分木一致」になっただけ。コード解析と LLM の節約が同じ原理に従う
> (→ `Lapilli/packages/llm-gateway/DESIGN.md`)。

## 3. 全体アーキテクチャ

```
┌── 静的層: コンテンツアドレス DAG ─────────────────────┐
│ Parse(tree-sitter) → Merkle-AST → 機構契約 → 機構カード │
│                          ↕ 3段リンカ                    │
│                       仕様クレーム                      │
│                    └─ KG 射影 (traceability クエリ)     │
└──────────────────────────┬──────────────────────────────┘
                           │ 同一 Anchor ID で縫合
┌──────────────────────────┴── 動的層: 実行トレース ──────┐
│ 静的ループ骨格抽出 → マーカー自動注入 → フレーム×機構トレース │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴── 配信・可視化層 ────────────┐
│ inAppBridge 相乗り → Web ビュー (タイムライン/光るグラフ/You are here) │
│              → AI 文脈束 (決定的・キャッシュ安定)         │
└─────────────────────────────────────────────────────────┘
```

すべてのノード・トレースは **Anchor ID = 正規化 Merkle ハッシュ** で識別され、層を跨いで結合する。

## 4. 静的層: コンテンツアドレス DAG

### 4.1 Merkle-AST

- `tree-sitter` で AST を抽出し、**各ノードに Merkle ハッシュ**を振る (ノード内容 + 子ハッシュ)。
- 仕様クレームやマーカーは、この **部分木ハッシュ (= Anchor ID) にアンカー**する。
- 関数を 1 個編集 → その部分木と祖先のハッシュだけ変化 → **依存する成果物だけ無効化**。
  スナップショット機構・差分索引機構は不要。Merkle 構造がそのままバージョニング兼無効化。

### 4.2 正規化ハッシュ (命中率の心臓)

最難所。素朴に AST をハッシュすると空白・改行・ローカル変数リネームで全部無効化する。

- **整形・コメントは無視**してハッシュ (構文木の形のみ)。
- ローカル変数名は **α 正規化** (位置インデックス化) してリネーム耐性を持たせる。
- 公開シンボル名・型は含める (意味が変わるため)。
- 何を無視し何を含めるかが命中率を決める。プロンプトキャッシュの「サイレント無効化犯」問題が
  *再帰的に* 出てくる領域。ここを設計の中心に置く。

### 4.3 機構契約 (Mechanic Contract)

機構 (skill / action / state machine / beat / system tick …) を **契約** として定義する。

- 契約 = 入力 / 出力 / 状態効果 のシグネチャ。
- コードが契約を満たすかを **conformance check** (真偽 + 根拠) で判定。類似度スコアではない。
- メカニクス検出が「当てもの」から「型検査」になる。1 機構 = 1 契約 = plugin (§7)。
- 緩さの設計が要る (硬すぎる契約は現実のコードを取りこぼす)。部分適合 + 信頼度を許容する。

### 4.4 機構カード (Mechanic Card)

キャッシュ・配信の **単位**。生コードでも生仕様でもなく、LLM が蒸留した正準サマリ。

- 内容: 「何の機構か / 契約 / 主要コードアンカー / 仕様参照 / 複雑度」。
- **下の code+spec の Merkle ハッシュをキーに 1 回だけ計算** (高い LLM パス) → content-addressed でキャッシュ。
- コード変更 → ハッシュ変化 → **影響した機構のカードだけ再計算**。
- この 1 個が **AI には高精度文脈束、人間には機構/複雑度サマリ** を兼ねる
  (= §1 の 2 用途が単一単位に統合される)。

### 4.5 コード ↔ 仕様 結合 (別設計書 Vinculum)

コードアンカーと仕様クレームの結合 (3 段リンカ / 硬化ループ / KG 射影 / 決定的束) は、
独立した設計書 **Vinculum** (`ergo/spec/tool/vinculum.md`) に分離した。
Anatomia の Anchor ID (§4.1-4.2) と機構カード (§4.4) を土台に、その上の結合層として定義する。

要点のみ:
- 結合は KG (Kuzu) を背骨に、embedding は信号に留める (キャッシュは決定的 retrieval が必須)。
- 明示 > 構造 > 意味の 3 段リンカ + 硬化ループ (使うほど決定的に)。
- 詳細・データモデル・留保は Vinculum 参照。

## 5. 動的層: 実行トレース

「今どのフレーム / どのソースにいるか」を監視し、静的 DAG に重ねる。

### 5.1 フレーム同期スコープ計測

ゲームはメインループ = 自然なフレーム境界を持つ。「今どのフレームか」= ループ毎にカウンタ、
「今どのソースか」= アクティブな zone スタック (各 zone が Anchor ID を持つ)。

| | C++ | C# / Unity |
|---|---|---|
| スコープ計測 | Tracy / Optick / Remotery 系 (フレーム整合・source 付き・低オーバヘッド) | ProfilerMarker / CustomSampler (Unity)、`EventSource` |
| 無改変サンプリング | サンプリング + DWARF/PDB で IP→source | EventPipe / dotnet-trace、ICorProfiler enter/leave |

### 5.2 マーカー自動注入

手置きは精密だが面倒、自動計測は楽だが機構を知らない。**静的解析が機構境界を知っている**ため両取り:

- 解析器が「機構カードの入口」を知る → **そこにだけマーカーをコード生成/注入**。
- マーカーは **静的 DAG と同じ Anchor ID** を埋め込む → 静的と動的が完全に噛み合う。
- 精密・低オーバヘッド・手置き不要・機構タグ付き。

### 5.3 静的ループ骨格 + 動的アニメート

- 静的: `main()` / tick から辿る call graph で「System tick の順序 = あるべきループ構造」を先に描く。
- 動的: 実トレースでそれを *アニメート*。設計 (静的) と実際 (動的) のズレが見える。

### 5.4 同一アンカー縫合

zone(Anchor ID) ↔ 機構カード を join。
「フレーム N でこの順に System が回った / どの機構がホットか」が、静的カードに紐付いて見える。

## 6. 配信・可視化層

### 6.1 Transport

- LUDIARS には既に配信路がある: `ergo_custos` のアプリ内 HTTP ブリッジ (`inAppBridge /stream`,
  port 17778) + Custos の WebRTC / Unity View 配信。**トレースをこの transport に相乗り**させ、
  ビューアを Web で出す。
- プロセス内リングバッファ → 別プロセス (ビューア) へ stream。低オーバヘッド。

### 6.2 ビュー

`render_pipeline` plugin の「静的 scan → Web 可視化」前例の *動的版*。

- **フレーム × 機構タイムライン**: 各フレームで実行された System/機構を順序・時間付きで。
- **光るグラフ**: 静的 DAG/KG を、現在アクティブな Anchor で点灯。
- **You are here**: 現在フレーム + アクティブ zone スタックのライブカーソル。

## 7. 拡張性 (3 種 plugin)

Ergo の plugin pack + `ERGO_PLUGIN_DIR` モデルに合わせる。

1. **言語フロントエンド** plugin: tree-sitter grammar 単位 (C++/C#/TS/Rust…)。
2. **機構オントロジー** plugin: ゲーム別/ドメイン別の機構語彙 = 「契約 + 検出ルール + カード要約テンプレ」。
   - 例: AdventureCube (`Skill→ActiveObject→Action` + beat + Shield/Melee)、
     ergo_character (`Brain/Check/Control/Effect`)、KuzuSurvivors (wave/pickup/upgrade)、
     Memoria 等は汎用 (CRUD/Pipeline/Adapter)。
3. **出力アダプタ** plugin: Web 可視化 / AI 文脈束 / 人間向けレポート。

コア (Parse/Model/Analyze) は不変、上記 3 点で各ゲーム・各言語へ広げる。

## 8. 複雑度メトリクス (game-aware)

cyclomatic だけでは不足。ゲームの複雑度は **システム間結合と状態空間**に出る。

- 機構オーバーラップ密度 (1 entity に触る機構数)
- 共有 state への fan-in
- 機構をまたぐ依存深さ
- (補助) cyclomatic / fan-in-out / 結合度

メトリクスはすべて DAG/KG 上のグラフ集計として算出 (機構整理と同じグラフの副産物)。
Memoria 肥大化も同じ尺度 (1 モジュールに機能が何個刺さっているか)。

## 9. キャッシュ設計

- **文脈束 = `f(query, DAG スナップショット)` の純関数**。安定ソートで決定的に組む。
- 同じ問い → 同じ束 → prefix 安定 → `cache_read` が効く。
- 変更時は Merkle で影響範囲だけ無効化 (差分索引と地続き)。
- 束の中も「不変 (機構サマリ/型定義) を前、可変 (今回の問い) を後ろ」に並べる
  → `@ludiars/llm-gateway` の `orderSegments` をそのまま使う。
- **本ツール自体がトークン節約の実装例**になる (前方一致 = 部分木一致の自己相似)。

## 10. RAG・代替の評価 (なぜこの構成か)

「キャッシュ機構」前提では、決定的 retrieval が必須。

| 方式 | 精度 | キャッシュ適性 | 機構/複雑度抽出 | 評価 |
|---|---|---|---|---|
| Embedding RAG | △ | **✕ (揺れる)** | ✕ | 曖昧な入口の補助のみ |
| 静的解析 DAG/KG | ◎ | **◎ (決定的)** | ◎ | **核** |
| リポマップ | ○ | ○ | △ | 軽量版・補助 |
| LSP 駆動 | ◎ | ○ | △ | 裏取りに使用 |
| エージェント検索 | ◎ | ○ | ✕ | leaf 本文取得に使用 |

- embedding-RAG は retrieval が揺れて prefix が揺れる → キャッシュと本質的に不適合。
- KG は結合の *保管・配信* に最適だが、結合の *生成* は 3 段リンカ (§4.5) 次第。KG 単体を
  「1 番」と見ると本丸 (リンカ) を見落とす。
- → コンテンツアドレス DAG を下層の真実、KG をその射影、embedding を信号、に役割分離。

## 14. 段階計画 (実装は別タスク。本書は設計のみ)

1. **PoC: Merkle-AST + 正規化ハッシュ** — 命中率の心臓を最初に検証 (1 言語 = C++)。
2. **機構契約 + カード** — 1 ゲーム (例: AdventureCube) の機構を契約化し、カードを content-addressed で生成。
3. **コード↔仕様リンカ (明示 + 構造)** — 硬化ループの批准フローまで。意味リンカは後。
4. **動的トレース** — 静的ループ骨格抽出 → マーカー自動注入 → フレーム×機構トレース。
5. **可視化** — inAppBridge 相乗り → Web ビュー。
6. **AI 文脈束アダプタ** — 決定的束を serve。

各段で「正規化ハッシュの命中率」「契約の取りこぼし率」「束の決定性」を計測する。

## 15. 非目標

- 完全自動の意味リンク (硬化ループで人間/CI 批准を前提)
- 全言語の即時対応 (言語フロントエンド plugin で漸進)
- 実行時の重い常時計測 (計測ビルドを分離。KS は Release 必須のため計測フラグ付き Release を用意)
- 本ツール内でのプロンプト圧縮モデル同梱 (インターフェース注入で外部化)

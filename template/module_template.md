# <モジュール名> モジュール定義

このファイルは新しい Ergo モジュール仕様書を書くときの雛形。
`spec/module/<name>.md` にコピーして内容を埋めること。

不要な節は **節ごと削除してよい**。空の見出しを残すと「未決定」なのか
「不要」なのかが読み手に伝わらないため、必ずどちらか決める。

---

## 概要

そのモジュールが何を解決するか、ホストアプリからどう使うか、他モジュール
との関係を 2〜4 パラグラフ程度で述べる。実装詳細ではなく **役割・境界**
の記述を優先する。

## カテゴリ

次のいずれかを 1 つ:

- UI
- ロジック
- システム
- ゲームオブジェクト

## 所属ドメイン

より細かい分類 (自由記述)。例: 「入力」「サウンド」「開発者ツール /
デバッグ」「描画」など。

## 必要なデータ

ランタイムで保持・参照する外部データ (ホストから受け取る設定、外部
ファイル、ネットワーク入出力、デフォルト値など) を箇条書き。

## 依存

- C++17 標準ライブラリのうち使う主要ヘッダ
- 他 Ergo モジュール (`ergo_foo` など)
- サードパーティ (Vulkan / Pictor / Threads など)
- テスト依存 (GoogleTest 互換 mini-gtest など)

## 変数

実装ファイル内部の主要な状態変数を列挙。ドキュメントとして重要なのは
**どんな状態を持つか** であって、フィールド名までの厳密一致は不要。

## 作業

### 入力

ホストから呼ばれる API、外部イベント、上流モジュールから届くメッセージ。

### 出力

ホストへ返す API の戻り値、下流モジュールへ送るメッセージ、副作用
(ログ出力・WS 送信・ファイル書き出しなど)。

### タスク

毎フレームの処理、スレッドの役割、初期化/終了シーケンスなど。
「誰がいつ何をするか」を読者が追えること。

### プラットフォーム別タスク

Windows / Linux / macOS / WebGL など、実装差分がある場合のみ記す。
差分が無ければこの節は削る。

## テスト

ユニットテストでカバーすべき観点を列挙。実装のディテールではなく
「これが守られていれば仕様通り」と言える観点に絞る。
